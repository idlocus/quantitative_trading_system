# Python 项目开发规则

## 1. 代码风格

### 1.1 PEP 8 规范
- 遵循 [PEP 8](https://peps.python.org/pep-0008/) 代码风格指南
- 使用 4 个空格进行缩进（不要使用制表符）
- 每行代码长度不超过 79 个字符
- 函数和类之间用两个空行分隔
- 函数内部逻辑块之间用一个空行分隔

### 1.2 导入语句
- 导入语句按以下顺序排列：
  1. 标准库导入
  2. 相关第三方库导入
  3. 本地应用/库特定导入
- 每组导入之间用空行分隔
- 使用绝对导入，避免相对导入
- 避免使用 `from module import *` 导入方式

## 2. 命名约定

### 2.1 变量和函数
- 变量名：使用小写字母，单词之间用下划线分隔（snake_case）
- 函数名：使用小写字母，单词之间用下划线分隔（snake_case）
- 常量：使用全大写字母，单词之间用下划线分隔（UPPER_SNAKE_CASE）

### 2.2 类和模块
- 类名：使用驼峰命名法（CamelCase）
- 模块名：使用小写字母，单词之间用下划线分隔（snake_case）
- 包名：使用小写字母，不使用下划线

### 2.3 私有成员
- 私有变量和方法：使用单下划线前缀（_private）
- 强私有变量和方法：使用双下划线前缀（__private）

## 3. 代码组织

### 3.1 项目结构
```
project_name/
├── README.md
├── setup.py
├── requirements.txt
├── .gitignore
├── project_name/
│   ├── __init__.py
│   ├── module1.py
│   ├── module2.py
│   └── utils/
│       ├── __init__.py
│       └── helper.py
└── tests/
    ├── __init__.py
    ├── test_module1.py
    └── test_module2.py
```

### 3.2 函数和类
- 函数长度应控制在合理范围内，通常不超过 50 行
- 每个函数应只负责一个明确的任务
- 类的设计应遵循单一职责原则
- 超过 20 行的代码应优先抽象为函数或模块

## 4. 文档要求

### 4.1 注释
- 函数和类必须使用文档字符串（docstring）
- 文档字符串应遵循 [PEP 257](https://peps.python.org/pep-0257/) 规范
- 复杂的逻辑应添加行内注释
- 注释应使用中文，清晰地解释代码的意图

### 4.2 README.md
- 项目简介
- 安装说明
- 使用方法
- 贡献指南
- 许可证信息

## 5. 测试规范

### 5.1 测试框架
- 使用 `pytest` 作为测试框架
- 测试文件应放在 `tests/` 目录下
- 测试文件名应以 `test_` 开头

### 5.2 测试覆盖
- 重构前确保测试覆盖
- 新功能必须添加相应的测试用例
- 目标测试覆盖率应不低于 80%

## 6. 错误处理

### 6.1 异常处理
- 使用 try-except 块捕获并处理异常
- 只捕获必要的异常类型，避免捕获所有异常
- 异常处理应包含具体的错误信息
- 避免使用异常进行流程控制

### 6.2 日志记录
- 使用 `logging` 模块进行日志记录
- 日志级别应合理设置（DEBUG, INFO, WARNING, ERROR, CRITICAL）
- 生产环境应避免使用 `print` 语句进行调试

## 7. 并发控制

### 7.1 线程安全
- 使用线程安全的并发控制机制
- 多线程操作共享资源时应使用锁
- 优先考虑使用 `threading` 模块的 `Lock`、`RLock` 等同步原语
- 对于复杂的并发场景，考虑使用 `concurrent.futures` 模块

### 7.2 异步编程
- 对于 I/O 密集型任务，考虑使用 `asyncio` 进行异步编程
- 异步函数应使用 `async` 和 `await` 关键字

## 8. 性能考虑

### 8.1 代码优化
- 避免不必要的计算和重复操作
- 使用适当的数据结构（如字典、集合）提高查找效率
- 对于大型数据集，考虑使用生成器节省内存

### 8.2 性能测试
- 对于性能关键的代码，应进行性能测试
- 使用 `timeit` 模块或 `cProfile` 进行性能分析

## 9. 安全性

### 9.1 代码安全
- 避免使用不安全的函数（如 `eval()`、`exec()`）
- 输入数据应进行验证和清理
- 密码等敏感信息应加密存储

### 9.2 依赖安全
- 定期更新依赖包以修复安全漏洞
- 使用 `pip-audit` 等工具检查依赖包的安全问题

## 10. 版本控制

### 10.1 Git 规范
- 使用 Git 进行版本控制
- 提交信息应清晰明了，遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范
- 分支管理应遵循合理的策略（如 Git Flow）

### 10.2 .gitignore
- 项目应包含合理的 `.gitignore` 文件
- 忽略不需要版本控制的文件（如编译产物、依赖包、日志文件等）

## 11. 依赖管理

### 11.1 包管理
- 使用 `pip` 进行包管理
- 项目应包含 `requirements.txt` 文件，列出所有依赖包及其版本
- 对于复杂项目，考虑使用 `pipenv` 或 `poetry` 进行依赖管理

### 11.2 虚拟环境
- 开发时应使用虚拟环境（如 `venv`）隔离依赖
- 避免在全局环境中安装项目依赖

## 12. 代码审查

### 12.1 审查流程
- 重构后必须通过代码审查
- 代码审查应关注代码质量、安全性、性能等方面
- 审查意见应及时反馈和处理

### 12.2 代码质量工具
- 使用 `flake8` 检查代码风格
- 使用 `pylint` 进行代码质量分析
- 使用 `mypy` 进行类型检查（如果使用类型注解）

## 13. 持续集成

### 13.1 CI/CD 配置
- 项目应配置持续集成（如 GitHub Actions、GitLab CI）
- CI 流程应包含代码风格检查、测试执行、构建等步骤
- 对于部署项目，应配置持续部署流程

## 14. 其他约定

- 代码应具有可移植性，避免依赖特定平台的特性
- 遵循小步重构原则，每次仅修改一个逻辑单元，修改后立即测试
- 保持代码的可读性和可维护性，优先考虑代码清晰度而非过度优化
- 定期进行代码清理，移除无用的代码和注释
